Overview

Built around the actual residency funnel

The product turns the PRD into a single operating surface: candidate intake, screening, interviews, decisions, and reporting connected under role-based access.

Objectives

Replace fragile spreadsheet ops

Centralize candidate records CVs, tests, evaluations, and status changes live in one batch-aware system.
Protect candidate PII Reviewers never see email or phone, and reviewer CV access is visibly watermarked.
Standardize interviews Assignments, rubric scores, minimum-comment rules, and read-only submissions.
Drive decisions with reporting Funnel metrics, university breakdowns, workload views, and exportable summaries.

Import Wizard

Guardrails before data lands

1. Upload CSV intake plus ZIP-based CV upload for one recruitment batch.
2. Column Mapping Map source headers to the candidate schema with import-time validation.
3. Preview & Validate Catch invalid GPA ranges, malformed email addresses, and duplicates by email.
4. Confirm Persist candidates, upload CVs, and log the import event to audit history.

Security

Role-based access is part of the product, not decoration

The backend contract assumes Supabase Auth, Row Level Security, protected CV access, and automatic audit logging via triggers.

Access Matrix

Who can see what

Super Admin

All candidate data, users, batches, audit logs, and policy-sensitive controls.

Coordinator

Candidate operations, imports, assignments, exports, and dashboards. No user management.

Reviewer

No email, no phone, no contact export, no CV download, only own evaluations writable.

Controls

Enforced by the stack

  • Google OAuth using pre-provisioned corporate accounts only
  • RLS on every table backing the recruitment workflow
  • Audit logs readable only by super admin
  • Watermarked reviewer CV viewer with right-click disabled in the real app
  • Trigger-based logging for candidates, assignments, and evaluations

UX Requirements

Operational polish

  • Responsive desktop and tablet layouts
  • Loading skeletons during data fetch
  • Confirmation dialogs for destructive actions
  • Toast feedback for success and error states
  • Friendly empty states and a dedicated 404 page
